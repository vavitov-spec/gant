# План: роли и права доступа

Статус: **план согласован, реализация не начата**. Обсуждение — 2026-07-06. Начинаем в следующей сессии.

## Задача

Сейчас в приложении нет реальной аутентификации: окно логина (`showAuth`/`authOverlay`) существует в разметке, но нигде не вызывается — «мёртвый» код. Доступ к правкам держится на одном общем пароле `EDIT_PASSWORD = '555'` (localStorage-флаг `canEdit`), без деления по пользователям.

Данные уже устроены как несколько проектов внутри одного JSON-документа: `data.projects[]`, переключение через `switchProject`. Но это просто массив в одной строке `app_state` — реальных таблиц/прав на проект нет.

Нужно: любой человек может зарегистрироваться → администратор проекта подключает пользователя к проекту с ролью (вручную или ссылкой-приглашением, в которой роль уже зашита) → 4 роли.

## Согласованные решения

### Роли и права
- **Администратор** (создатель проекта) — полный доступ, включая управление участниками/приглашениями.
- **Участник проекта** — читает и редактирует всё в проекте (как сейчас работает редактирование в режиме `canEdit`).
- **Подрядчик** — видит весь график (с фильтром «показать только мои работы» — использовать существующий фильтр по подрядчику), но факт может вносить **только по своим работам**. Обеспечивается не интерфейсом, а сервером (см. ниже).
- **Заказчик** — только просмотр (диаграмма, статистика, PDF), без пароля и без настроек.

### Приглашения
- Одна ссылка = один человек, **одноразовая** (после использования — сгорает).
- В ссылке зашиты: проект + роль (+ для Подрядчика — имя подрядчика для сопоставления с работами).
- Если человек не зарегистрирован — сначала регистрируется/входит, затем автоматически подключается к проекту с этой ролью.

### Вход
- Email + пароль через Supabase Auth (обычная регистрация). Без magic-link (не хотим зависеть от настройки email-провайдера в Supabase).
- Старый общий пароль редактирования `555` — **убрать полностью**, права полностью определяются ролью.

### Ключевая техническая находка
Всё приложение сохраняется **целиком одним документом**: `buildPayload()` собирает все проекты/работы → один `UPDATE app_state SET data = ...` (construction_gantt.html:3710-3751). Приём с сервера (`applyRemote` → `applyPayload`) тоже полностью заменяет состояние, без слияния по полям (construction_gantt.html:3885-3893, 3962+).

Следствие: обычная RLS-политика (на уровне строки) не может ограничить запись «только своя работа внутри JSON» — она видит только «можно писать в строку целиком или нет». А если дать Подрядчику право полной перезаписи документа (как Участнику), это ещё и создаёт **реальный риск потери факта** при одновременном вводе: два подрядчика, редактирующие разные работы почти одновременно, могут затереть правки друг друга — тот, чей снимок документа старше, перезапишет уже сохранённую правку другого (последний сохранивший побеждает, слияния нет).

**Решение:** для Подрядчика — отдельная серверная функция (RPC) `submit_fact(project_id, task_id, period, value)`, которая:
- точечно патчит факт внутри JSON (`jsonb_set` на нужный путь, найденный по id работы, а не по индексу массива),
- на сервере проверяет, что вызывающий — подрядчик именно этой работы,
- не даёт Подрядчику права на общий путь `cloudSave` (полную перезапись документа) — RLS блокирует полный `UPDATE app_state` для роли «Подрядчик».

Это же попутно решает проблему гонки при одновременной записи: два RPC-вызова к разным работам патчат разные ветки JSON и не мешают друг другу.

## Что нужно построить (план работ)

1. **Схема БД в Supabase**
   - `projects` (id, name, created_by, created_at)
   - `project_members` (project_id, user_id, role, contractor_name — для роли «Подрядчик», сопоставление с текстовым полем работы `t.contractor`)
   - `project_invites` (token, project_id, role, contractor_name, created_by, expires_at, used_at)
   - Миграция текущих данных: строка `app_state` (id=1, ЖК «Северная Звезда») получает запись в `projects`, текущий пользователь (ksandrblog@gmail.com) становится Администратором в `project_members`.

2. **RLS-политики** на `app_state` / `app_state_history`:
   - SELECT — только участникам проекта (`exists (... project_members ...)`).
   - UPDATE (полная перезапись документа) — только роли Администратор/Участник проекта.
   - Подрядчик и Заказчик — SELECT есть, полного UPDATE нет.

3. **RPC-функции** (Postgres, `SECURITY DEFINER`):
   - `submit_fact(project_id, task_id, period, value)` — точечная запись факта Подрядчиком, с проверкой `contractor_name` на сервере.
   - `accept_invite(token)` — проверяет токен (не использован, не истёк), создаёт запись в `project_members`, гасит токен.

4. **Аутентификация**: оживить `authOverlay`/`showAuth` — реальные регистрация и вход (email+пароль). Сейчас код существует, но не вызывается нигде. Убрать `EDIT_PASSWORD`/`canEdit`-логику, заменить проверками роли.

5. **UI администратора**: вкладка «Участники» в Настройках —
   - список участников проекта с ролями (изменить роль / удалить участника — только Администратору);
   - кнопка «Создать ссылку-приглашение» (выбор роли, для Подрядчика — выбор/ввод имени подрядчика) → генерирует ссылку вида `.../?invite=<token>`, копирование в буфер;
   - список неиспользованных приглашений с возможностью отозвать.

6. **Обработка ссылки-приглашения** при загрузке (`?invite=<token>` в URL): если нет сессии — сначала форма регистрации/входа; после входа — вызов `accept_invite`, затем переход в нужный проект.

7. **UI для Подрядчика**: использовать существующий фильтр по подрядчику, по умолчанию включать «только мои работы»; поля ввода факта для чужих работ — не кликабельны; для своих — вызывают `submit_fact` вместо обычного `saveState()`.

## Этап 6 (план, не начат): компании и точечное назначение подрядчиков

Статус: **дизайн согласован 2026-07-09, реализация не начата**. Заменяет/расширяет текущий `contractor_name` (свободный текст в `project_members`) на нормальную модель компаний. Начинать после завершения Этапа 5 (живая проверка текущей ролевой модели).

### Задача (с формулировки пользователя)
- Любой пользователь имеет аккаунт и может быть сотрудником нескольких компаний.
- Компании — отдельная сущность, ключевой классификатор — **ИНН**.
- Когда пользователь создаёт график — он становится РП (руководитель проекта) для этого графика (это и есть роль `admin`, переименования не требуется, только смысл уточняется).
- РП может привязать подрядчика (компанию по ИНН) к разделу, подразделу или конкретной работе, и выбрать из сотрудников этой компании — ответственного (роль `contractor` в программе), причём **точечно**, а не на весь график.
- У каждого пользователя есть системный ID (короткий, человекочитаемый), по которому его тоже можно найти и привязать к работе/разделу напрямую. Если у найденного пользователя одна компания — она подставляется автоматически; если несколько — РП выбирает, от какой компании он участвует в этом графике.

### Решение по управлению компанией (согласовано с пользователем)
**Модель «владелец компании»**: первый, кто зарегистрировал компанию по ИНН в системе, становится её владельцем (аналогично тому, как сейчас работает `admin` проекта). Владелец сам добавляет/удаляет сотрудников компании. РП при назначении подрядчика выбирает уже существующего сотрудника компании из списка; если нужного человека там ещё нет — отправляет ему приглашение в компанию (тем же механизмом токен-ссылок, что уже есть у `project_invites`).

### Новые таблицы
- **`companies`** — `id`, `inn` (уникальный, с проверкой контрольной суммы ИНН на стороне клиента/RPC), `name`, `owner_user_id`, `created_at`. РП может создать «пустышку» (только ИНН+название), если компании ещё нет в системе — владелец назначится позже через приглашение (первый принявший приглашение в пустую компанию становится её владельцем).
- **`company_members`** — `company_id`, `user_id`, `added_at`. Владелец компании — тоже строка в этой таблице (создаётся автоматически вместе с компанией).
- **`company_invites`** — `token`, `company_id`, `created_by`, `used_at` — приглашение сотрудника в компанию, зеркалирует существующий `project_invites`.
- **`profiles`** — `user_id` (PK, = `auth.users.id`), `public_id` (bigserial, человекочитаемый номер для поиска), `email` (денормализовано, т.к. `auth.users` недоступна с фронта — та же причина, по которой уже денормализован `user_email` в `project_members`). Создаётся автоматически при регистрации (триггер или первый вход).
- **`work_assignments`** — `id`, `project_id` (= `app_state_id`/`projects.id`), `scope_type` (`section` / `subsection` / `task`), `scope_id` (id узла графика — нужно проверить, что у разделов/подразделов такой стабильный id уже есть в JSON, не только у задач), `company_id`, `responsible_user_id`, `assigned_by`, `assigned_at`. Это и есть точечная привязка подрядчика — заменяет `contractor_name`.

### Как это стыкуется с уже реализованным
- Роль `contractor` в `project_members` сохраняется — она даёт человеку вообще доступ войти в график. А **что именно** он может редактировать, теперь определяет не текстовое совпадение `t.contractor`, а факт наличия строки в `work_assignments` на него.
- RPC `submit_fact_days` нужно расширить: для роли `contractor` проверять не только «участник этого проекта», а «есть ли `work_assignments`, чей `scope` покрывает именно этот `task_id`» (с учётом вложенности: если подрядчик назначен на раздел — это покрывает все работы внутри раздела). Это потребует на сервере знать, к какому разделу/подразделу принадлежит задача — нужно свериться со схемой JSON перед реализацией (Этап 6.3).
- Поиск пользователя по `public_id` — альтернативный (не через компанию) способ найти человека для назначения; итог всё равно сводится к выбору его компании (авто, если одна) для контекста конкретного графика.
